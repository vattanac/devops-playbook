Linux Fundamentals
Shell, filesystem, processes, permissions
▾

Why Linux?

Kubernetes runs on Linux. Every node in a cluster is a Linux machine. You'll SSH into nodes, read logs, debug containers — all requiring Linux skills.

Think of Linux as the foundation of a building. You can't build floors (containers) or a whole skyscraper (Kubernetes cluster) without a solid foundation.

Essential Commands

# Navigation & files
ls -la /var/log          # List files with details
cd /etc/kubernetes        # Change directory
cat /etc/os-release       # View file content
grep -r "error" /var/log  # Search inside files

# Process management
ps aux | grep kubelet     # Find kubelet process
systemctl status kubelet  # Check service status
journalctl -u kubelet -f  # Follow service logs

# Networking
ip addr show              # Show IP addresses
ss -tulnp                 # Show open ports
curl http://localhost:8080 # Make HTTP request

# Permissions
chmod 600 ~/.kube/config  # Set file permissions
chown user:group file     # Change ownership

Key Concepts to Master

  • Filesystem hierarchy — Understand /etc, /var, /proc, /sys
  • Process management — How processes start, stop, and communicate (signals)
  • Networking basics — IP addresses, ports, DNS resolution, iptables
  • Systemd — Service management (kubelet runs as a systemd service)
  • SSH — Remote access to nodes

Networking Essentials
TCP/IP, DNS, HTTP, load balancing

Why Networking Matters

Kubernetes is fundamentally a networking system. Pods talk to pods, services route traffic, ingress handles external requests. Without networking knowledge, you'll be lost.

The OSI Model (Simplified)

Critical Concepts

  • IP Addresses — Every Pod gets its own IP. Nodes have IPs. Services have virtual IPs (ClusterIP).
  • DNS — K8s has built-in DNS (CoreDNS). Services are reachable at my-service.my-namespace.svc.cluster.local
  • Ports — containerPort, targetPort, port, nodePort — each means something different
  • Load Balancing — kube-proxy distributes traffic across healthy Pods

Docker & Containers
Images, containers, Dockerfile, registries
▾

What is a Container?

A container is a lightweight, isolated process that packages your application with all its dependencies. Unlike a VM, it shares the host OS kernel — making it fast and efficient.

A VM is like renting an entire apartment (own kitchen, bathroom, everything). A container is like a co-working desk — you get your own space but share the building's utilities.

Core Docker Commands

# Build an image from a Dockerfile
docker build -t myapp:1.0 .

# Run a container
docker run -d -p 8080:80 --name web myapp:1.0

# List running containers
docker ps

# View logs
docker logs -f web

# Execute a command inside
docker exec -it web /bin/sh

# Push to a registry
docker tag myapp:1.0 registry.example.com/myapp:1.0
docker push registry.example.com/myapp:1.0

Example Dockerfile

Dockerfile
# Multi-stage build for a Node.js app

# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

# Stage 2: Runtime (smaller image)
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
USER node
CMD ["node", "dist/server.js"]

Always use multi-stage builds to keep your images small. A smaller image means faster deployments and smaller attack surface.

YAML Basics

Every Kubernetes resource is defined in YAML. Master it, and you'll read/write K8s configs effortlessly.

YAML
# Key-value pairs
name: my-app
version: "2.0"     # Strings
replicas: 3          # Numbers
enabled: true        # Booleans

# Nested objects (use 2-space indent!)
metadata:
  name: my-app
  labels:
    app: web
    tier: frontend

# Lists
containers:
  - name: web
    image: nginx:1.27
  - name: sidecar
    image: fluentd:latest

# Multi-line strings
description: |
  This is a multi-line
  string that preserves
  newlines.

# Multi-document file
---
kind: Service
---
kind: Deployment

YAML is indentation-sensitive. Always use spaces (not tabs) and be consistent with 2-space indentation. A single wrong indent will break your manifest.

What is Kubernetes?
Container orchestration explained
▾

The Problem Kubernetes Solves

Imagine you have 50 microservices running in containers. You need to: deploy them across servers, restart them if they crash, scale them under load, update them without downtime, and route traffic between them. Doing this manually? Impossible. Kubernetes automates all of it.

Kubernetes is like an air traffic controller at an airport. Planes (containers) need to land, take off, refuel, and park. The controller (K8s) makes sure everything runs smoothly, no planes crash, and if a runway (server) is busy, it reroutes traffic.

Key Features

  • Self-healing — Restarts crashed containers, replaces unresponsive ones
  • Auto-scaling — Adds/removes Pods based on CPU, memory, or custom metrics
  • Rolling updates — Deploy new versions with zero downtime
  • Service discovery — Built-in DNS and load balancing
  • Secret management — Securely store passwords, tokens, and certificates
  • Declarative config — You describe the desired state; K8s makes it happen
Declarative vs Imperative

Imperative: "Create 3 nginx pods, then create a service, then…" (you tell K8s each step).
Declarative: "I want 3 nginx pods behind a service" (you describe the end state, K8s figures out how to get there). Kubernetes favors the declarative approach.

Pods — The Smallest Unit
Understanding the basic building block
▾

What is a Pod?

A Pod is the smallest deployable unit in K8s. It wraps one or more containers that share the same network namespace (same IP), storage volumes, and lifecycle.

If a container is a person, a Pod is a shared apartment. Roommates (containers) share the same address (IP), kitchen (volumes), and front door (network). They can talk to each other via localhost.

Your First Pod

pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-first-pod
  labels:
    app: hello
    environment: learning
spec:
  containers:
    - name: web
      image: nginx:1.27-alpine
      ports:
        - containerPort: 80
      resources:
        requests:
          memory: "64Mi"
          cpu: "100m"
        limits:
          memory: "128Mi"
          cpu: "250m"
# Create the pod
kubectl apply -f pod.yaml

# Check it's running
kubectl get pods
# NAME           READY   STATUS    RESTARTS   AGE
# my-first-pod   1/1     Running   0          10s

# Access it locally
kubectl port-forward my-first-pod 8080:80
# Now visit http://localhost:8080

# View details
kubectl describe pod my-first-pod

# Delete it
kubectl delete pod my-first-pod

PhaseDescription

Multi-Container Pod (Sidecar Pattern)

sidecar-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-with-logging
spec:
  containers:
    - name: web          # Main application
      image: nginx:1.27
      volumeMounts:
        - name: logs
          mountPath: /var/log/nginx
    - name: log-shipper  # Sidecar: ships logs
      image: fluentd:latest
      volumeMounts:
        - name: logs
          mountPath: /var/log/nginx
          readOnly: true
  volumes:
    - name: logs
      emptyDir: {}

Namespaces divide a cluster into isolated virtual clusters. They're perfect for separating environments (dev/staging/prod) or teams.

Deployments & ReplicaSets
The standard way to run stateless apps
▾

What is a Deployment?

A Deployment manages a ReplicaSet, which manages Pods. It's the most common way to run stateless applications. It provides rolling updates, rollbacks, and scaling.

Deployment └── ReplicaSet (v2 - current) │ ├── Pod-1 │ ├── Pod-2 │ └── Pod-3 └── ReplicaSet (v1 - previous, 0 replicas) └── (kept for rollback)

Complete Deployment Example

deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
  labels:
    app: webapp
spec:
  replicas: 3                     # Run 3 identical pods
  selector:
    matchLabels:
      app: webapp                 # Must match template labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1               # 1 extra pod during update
      maxUnavailable: 0         # 0 pods unavailable during update
  template:                         # Pod template
    metadata:
      labels:
        app: webapp
    spec:
      containers:
        - name: webapp
          image: myapp:2.0.0
          ports:
            - containerPort: 8080
          env:
            - name: NODE_ENV
              value: "production"
          resources:
            requests:
              cpu: "200m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          readinessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 20

Deployment Operations

# Deploy
kubectl apply -f deployment.yaml

# Watch rollout
kubectl rollout status deployment/webapp

# Scale
kubectl scale deployment/webapp --replicas=5

# Update image (triggers rolling update)
kubectl set image deployment/webapp webapp=myapp:3.0.0

# View rollout history
kubectl rollout history deployment/webapp

# Rollback to previous version
kubectl rollout undo deployment/webapp

# Rollback to specific revision
kubectl rollout undo deployment/webapp --to-revision=2

StatefulSets
For databases, queues, and stateful apps
▾

When to Use StatefulSets

Use when your app needs: stable network identity, persistent storage per pod, or ordered deployment/scaling. Common use cases: databases (PostgreSQL, MySQL), message queues (Kafka, RabbitMQ), distributed systems (Elasticsearch, ZooKeeper).

StatefulSet vs Deployment

FeatureDeploymentStatefulSet
Pod namesRandom (webapp-7d9f8)Ordered (db-0, db-1, db-2)
StorageShared or ephemeralUnique PVC per pod
Scaling orderRandomSequential (0→1→2)
DNSService onlyEach pod gets a DNS name
statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: "postgres"       # Headless service name
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          ports:
            - containerPort: 5432
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:          # Each pod gets its own PVC
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: "10Gi"

Pod DNS names: postgres-0.postgres.default.svc.cluster.local, postgres-1.postgres..., etc.

DaemonSet — Run on Every Node

Ensures a copy of a Pod runs on every node (or selected nodes). Perfect for: log collectors, monitoring agents, network plugins.

daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      containers:
        - name: node-exporter
          image: prom/node-exporter:latest
          ports:
            - containerPort: 9100

Job — Run to Completion

job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  backoffLimit: 3           # Retry 3 times on failure
  ttlSecondsAfterFinished: 300  # Clean up after 5 min
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: myapp:2.0
          command: ["python", "manage.py", "migrate"]

CronJob — Scheduled Tasks

cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"      # Every day at 2 AM
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: postgres:16
              command:
                - /bin/sh
                - -c
                - "pg_dump -h postgres -U admin mydb > /backup/db-$(date +%F).sql"

Why Services?

Pods are ephemeral — they get new IPs every time they restart. A Service provides a stable endpoint (DNS name + virtual IP) that routes traffic to healthy Pods.

┌─────────┐ │ Client │ └────┬────┘ │ ┌──────▼──────┐ │ Service │ Stable IP: 10.96.0.100 │ "webapp" │ DNS: webapp.default.svc └──┬───┬───┬──┘ │ │ │ load balanced ┌────▼┐ ┌▼──┐ ┌▼────┐ │Pod 1│ │Pod│ │Pod 3│ │:8080│ │ 2 │ │:8080│ └─────┘ └───┘ └─────┘

Service Types

TypeAccessible FromUse Case
ClusterIPInside cluster onlyInternal services (database, cache)
NodePortOutside via node IP:portDevelopment, testing
LoadBalancerExternal via cloud LBProduction (cloud providers)
ExternalNameDNS aliasMap to external services

Ingress & Ingress Controllers
HTTP routing, TLS termination, virtual hosts
▾

What is Ingress?

Ingress manages external HTTP/HTTPS access to services. It provides URL-based routing, SSL termination, and virtual hosting — all through a single entry point.

Internet │ ▼ ┌──────────────────────────────────┐ │ Ingress Controller │ │ (nginx / traefik) │ ├───────────┬──────────┬───────────┤ │ │ │ │ │ /api/* │ /app/* │ /docs/* │ │ │ │ │ │ │ │ │ ▼ │ ▼ │ ▼ │ │ api-svc │ app-svc │ docs-svc │ └───────────┴──────────┴───────────┘
ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - myapp.example.com
      secretName: myapp-tls
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-service
                port:
                  number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 80

Network Policies
Firewall rules for pod-to-pod traffic
▾

Why Network Policies?

By default, all pods can talk to all other pods. Network Policies restrict this — like a firewall for your cluster. Essential for security.

network-policy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-to-db
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database           # Apply to database pods
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api             # Only api pods can connect
      ports:
        - protocol: TCP
          port: 5432             # Only on PostgreSQL port

---
# Default deny all ingress traffic in a namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}            # Applies to ALL pods
  policyTypes:
    - Ingress
    - Egress

Start with a default-deny policy, then explicitly allow the traffic you need. This follows the principle of least privilege.

Storage

Persistent data in Kubernetes: Volumes, PersistentVolumes, PersistentVolumeClaims, and StorageClasses.

💿
Volumes & Persistent Storage
PV, PVC, StorageClass, CSI drivers
▾

The Storage Problem

Containers are ephemeral — when a pod dies, its filesystem is gone. Volumes allow data to survive pod restarts and be shared between containers.

Storage Hierarchy

StorageClass (defines HOW storage is provisioned) │ ▼ PersistentVolume (PV) ←── actual disk (NFS, EBS, GCE PD) │ ▼ (bound) PersistentVolumeClaim (PVC) ←── pod's request for storage │ ▼ (mounted) Pod → Container → /data/

Volume Types

TypePersists?Use Case
emptyDirPod lifetimeTemp files, cache, inter-container sharing
hostPathNode lifetimeDevelopment only (not safe)
PersistentVolumeClaimIndependentDatabases, file storage
configMap / secretConfig changeConfiguration files
storage-class.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com   # AWS EBS CSI driver
parameters:
  type: gp3
  fsType: ext4
reclaimPolicy: Retain          # Keep data after PVC deleted
volumeBindingMode: WaitForFirstConsumer

---
pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  storageClassName: fast-ssd
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: "50Gi"

---
pod-with-pvc.yaml
apiVersion: v1
kind: Pod
metadata:
  name: postgres
spec:
  containers:
    - name: postgres
      image: postgres:16
      volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: postgres-data

Resource Management

Requests vs Limits

RequestsLimits
WhatGuaranteed minimumMaximum allowed
SchedulingUsed by schedulerNot used for scheduling
EnforcementSoft (pod won't start if not enough)Hard (pod killed if exceeded)
resources:
  requests:
    cpu: "250m"        # 0.25 CPU cores (guaranteed)
    memory: "256Mi"    # 256 MiB RAM (guaranteed)
  limits:
    cpu: "1000m"       # 1.0 CPU cores (throttled above this)
    memory: "512Mi"    # 512 MiB RAM (OOM killed above this)

ResourceQuota per Namespace

resource-quota.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: development
spec:
  hard:
    requests.cpu: "10"
    requests.memory: "20Gi"
    limits.cpu: "20"
    limits.memory: "40Gi"
    pods: "50"
    persistentvolumeclaims: "10"

RBAC — Role-Based Access Control
Who can do what in your cluster
▾

RBAC Components

ServiceAccount / User / Group │ │ bound via ▼ RoleBinding / ClusterRoleBinding │ │ references ▼ Role / ClusterRole (defines permissions: verbs + resources)
rbac.yaml
# 1. Create a ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: deploy-bot
  namespace: production
---
# 2. Define permissions (Role = namespace-scoped)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: deployer
  namespace: production
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "update", "patch"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
---
# 3. Bind ServiceAccount to Role
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: deploy-bot-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: deploy-bot
    namespace: production
roleRef:
  kind: Role
  name: deployer
  apiGroup: rbac.authorization.k8s.io

Check Permissions

# Can I create deployments?
kubectl auth can-i create deployments

# Can deploy-bot list pods in production?
kubectl auth can-i list pods \
  --as=system:serviceaccount:production:deploy-bot \
  -n production

Pod Security & Best Practices
SecurityContext, PodSecurity Standards, hardening
▾

Hardened Pod Example

secure-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  securityContext:
    runAsNonRoot: true          # Never run as root
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: myapp:2.0@sha256:abc123...  # Pin by digest
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]          # Drop ALL Linux capabilities
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
  automountServiceAccountToken: false  # Don't mount SA token

Security Checklist

  • Run containers as non-root user
  • Set readOnlyRootFilesystem: true
  • Drop all capabilities, add only what's needed
  • Use image digest pinning (not just tags)
  • Set resource requests and limits
  • Enable network policies (default deny)
  • Use Pod Security Admission (restricted profile)
  • Scan images for vulnerabilities (Trivy, Snyk)
  • Don't mount service account tokens unless needed
  • Use external secret managers for production

What is GitOps?

GitOps means your Kubernetes manifests live in a Git repo, and a tool (ArgoCD, Flux) automatically syncs them to the cluster. Push a change to Git → cluster updates itself.

Developer → Git Push → GitHub Repo │ ArgoCD watches │ ▼ Kubernetes Cluster (auto-syncs to match Git)

ArgoCD Application

kind: Application
metadata:
  name: webapp
  namespace: argocd
spec:
  project: default
  source:

    targetRevision: main
    path: apps/webapp/overlays/production
  destination:

What is Helm?

Helm is a package manager for Kubernetes. A Helm chart is a bundle of templated K8s manifests. You install apps with a single command, customize via values, and manage upgrades/rollbacks.

# Install a chart

  --namespace database --create-namespace \
  --set auth.postgresPassword=secretpass \
  --set primary.persistence.size=50Gi

# List releases
helm list -A

# Upgrade
helm upgrade my-release bitnami/postgresql \
  --set primary.persistence.size=100Gi

# Rollback
helm rollback my-release 1

# Uninstall
helm uninstall my-release -n database

Why Kustomize?

Kustomize lets you customize YAML without templates. You have a "base" set of manifests and apply "overlays" for each environment. Built into kubectl.

│   ├── kustomization.yaml
│   ├── deployment.yaml
│   └── service.yaml

│   │   └── kustomization.yaml
│   ├── staging/
│   │   └── kustomization.yaml
│   └── production/
│       ├── kustomization.yaml

  - name: myapp

kubectl apply -k overlays/production/

hpa.yaml

kind: HorizontalPodAutoscaler

  name: webapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1

        target:
          type: Utilization

      stabilizationWindowSeconds: 300  # Wait 5 min before scaling down
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60

Service Mesh (Istio)

What is a Service Mesh?

A service mesh adds a sidecar proxy (Envoy) to every pod. This proxy intercepts all network traffic, enabling: mutual TLS encryption, traffic splitting (canary deployments), circuit breaking, distributed tracing, and retry policies — all without changing your application code.

┌─────────────────────┐ ┌─────────────────────┐ │ Pod A │ │ Pod B │ │ ┌─────┐ ┌──────┐ │ │ ┌──────┐ ┌─────┐ │ │ │ App │←→│Envoy │←──mTLS──→│Envoy │←→│ App │ │ │ └─────┘ └──────┘ │ │ └──────┘ └─────┘ │ └─────────────────────┘ └─────────────────────┘ ▲ ▲ │ Istiod (control plane) │ └────────────────────────┘

Canary Deployment with Istio

Production Checklist & Patterns
Everything you need before going to production
▾

Production Readiness Checklist

  1. High Availability — Multi-node control plane (3+ etcd nodes), pods spread across availability zones using topologySpreadConstraints
  2. Resource Management — Set requests and limits on every container. Use LimitRanges and ResourceQuotas per namespace.
  3. Health Checks — Configure readinessProbe, livenessProbe, and startupProbe for every container.
  4. Security — RBAC, NetworkPolicies, PodSecurity (restricted), image scanning, secrets encryption at rest.
  5. Observability — Prometheus metrics, centralized logging (Loki/EFK), distributed tracing (Jaeger), alerting rules.
  6. Backup & DR — etcd snapshots, Velero for cluster backup, PV snapshots, multi-region strategy.
  7. GitOps — All manifests in Git, ArgoCD/Flux for automated deployment, PR-based changes.
  8. Scaling — HPA for pods, Cluster Autoscaler for nodes, PDB (PodDisruptionBudget) to prevent outages.
  9. Networking — Ingress with TLS (cert-manager), DNS management (external-dns), rate limiting.
  10. Upgrades — Cluster version upgrade strategy, test in staging first, use PDBs during node drains.

Pod Disruption Budget

pdb.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: webapp-pdb
spec:
  minAvailable: 2            # Always keep at least 2 pods running
  selector:
    matchLabels:
      app: webapp
